How Deal Hunter works

← back to the dashboard · source on GitHub ↗

Every morning a GitHub Action reads the day's deal-flow emails, extracts candidate properties, enriches the survivors with public data, underwrites them against a personalized buy box, and publishes the verdicts here. No servers, no paid data — the whole system is a cron job, a Python package, and a static page.

The pipeline

1
Ingest

Gmail API with a read-only OAuth scope pulls the last 48 hours from five deal-flow senders (an agent covering all deal types + four STR-focused newsletters).

↓
2
Extract

Claude parses messy newsletter HTML into structured candidates: address, price, and the seller's claimed numbers — which are tagged seller-claimed and never trusted directly.

↓
3
Dedupe

Normalized-address hashing against the SQLite database, so a property pitched by three newsletters is analyzed once.

↓
4
Kill filter

Deterministic first-pass checks that spend zero API calls: price ceilings the capital can't reach, confirmed non-destination markets, deals that fail the seller's own optimistic math. Most emailed deals die here, for free.

↓ survivors only
5
Enrich

RentCast free tier (tax records, rent estimates, valuations), STR revenue estimates via web search, county appraisal-district tax records, live investment mortgage rates, and a short-term-rental ordinance check that is never auto-cleared.

↓
6
Score

Pure, unit-tested Python: cap rate, cash-on-cash, DSCR, GRM, break-even occupancy → three tier-specific buy boxes → hard disqualifiers → tax-strategy triggers → PASS / BORDERLINE / FAIL plus a 0–100 composite score.

↓
7
Publish

A ranked digest lands as a GitHub issue (full deal cards for the top 3), the database is committed, and this dashboard redeploys.

Design decisions

  • LLM only where language is messy; code everywhere math must be right. Claude does parsing and web-search synthesis. Every metric, threshold comparison, and verdict is deterministic, unit-tested Python — a scoring engine you can't reproduce is a vibe, not an underwriting tool.
  • Judgment calls are flagged, never automated. Borderline destination-market status, which BORDERLINE deals to act on, rehab estimates, and final STR-legality verification are surfaced explicitly and left to the human. The pipeline makes the judgment call cheap; it doesn't make it.
  • Personal financial data never enters the repo. Scoring reads a gitignored private profile; the repo ships a sanitized example so it runs for anyone. This page shows verdicts and metrics — never income, capital, or tax numbers.
  • The strategy itself is versioned. The investment framework is markdown in the repo. When an assumption proves wrong, the fix is a commit with a diff history.

Why v2 exists: the project started as parallel AI-partner experiments that drifted into contradiction — different LTV ceilings, tax law that had been superseded (the OBBBA made 100% bonus depreciation permanent), different capital plans. An audit established precedence, filled the gaps neither version covered (a first-pass kill stage, STR operating-expense defaults), and the reconciled framework became the single source of truth the pipeline enforces.
